# Compatibility Notes

Dynesty.jl aims to align behavior, algorithms, numerical semantics, and edge
cases with Python dynesty unless an intentional Julia-native difference is
documented here.

| Area | Python behavior | Julia behavior | Reason | Affected tests |
| --- | --- | --- | --- | --- |
| Mutating APIs | Methods such as `run_nested` mutate sampler state. | Public Julia API uses bang forms such as `run_nested!`, `checkpoint!`, and `add_live_points!`; no-bang mutating aliases are not exported. | Julia convention makes mutation explicit. | `test/test_api_surface.jl`, sampler tests |
| Dimension indices | Periodic and reflective dimensions use Python 0-based indices. | Julia APIs use 1-based indices and do not expose Python index conversion helpers. | Preserve Julia indexing semantics and avoid implicit off-by-one conversions. | `test/test_api_surface.jl`, Stage 4 periodic/reflective tests and fixtures |
| User input arrays | Python dynesty defensively copies arrays before calling user functions. | Dynesty.jl defaults to `copy_inputs=false`; `copy_inputs=true` requests Python-like safety. | Performance-first Julia hot paths with documented read-only input contract. | Stage 5 likelihood/prior-transform tests |
| Result blob field | Python uses `res.blob`. | Julia results use `res.blobs`. | Julia field should describe the collection of per-sample blobs. | Stage 1 results tests and Stage 5 blob tests |
| Result post-processing fields | Python exposes `information`; some helper internals replace the non-public `h` key without changing `information`. | Julia stores `information` as the public field and allows `h` as an internal compatibility field when present. `reweight_run` follows the upstream `h` replacement behavior. | Preserve deterministic fixture compatibility while keeping the public Julia field stable. | `test/test_results_postprocess.jl` |
| Results compatibility aliases | Python fixture/result keys include `blob`, `samples_bound`, and `batch`. | Julia public `Results` uses `blobs`, `boundidx`, and `samples_batch`; Python fixture schema conversion belongs in test helpers, not public API. | Keep the public schema Julia-native while still allowing tests to compare Python fixtures. | `test/test_api_surface.jl`, `test/test_results.jl` |
| Pool semantics | Python exposes a `Pool` wrapper and `use_pool` controls for prior transform, log-likelihood, proposal evolution, bound update, and dynamic stopping functions. | Julia exposes `SerialMapBackend`, `ThreadedMapBackend`, `DistributedMapBackend`, and `ParallelPolicy` with sampler-level `parallel`, `map_backend`, `queue_size`, `parallel_policy`, and `proposal_scheduler=:batch/:async/:auto` keywords. `ParallelPolicy` uses Julia-native fields: `initialization`, `proposals`, `bounds`, and `stopping`. The backend is used by default for initial live-point evaluation and, when `queue_size > 1`, for the main proposal/evolve queue. Built-in ellipsoid and friends bound bootstrap work can use the backend when `ParallelPolicy(bounds=true)` and `bootstrap > 0`; the default is serial. The default dynamic `stopping_function` can use the backend for Monte Carlo error realizations when `ParallelPolicy(stopping=true)` and `n_mc > 1`; custom stop functions keep the serial call path unless they provide their own mapping behavior. `proposal_scheduler=:async` is currently threaded-only, and `:auto` falls back to batch for serial, distributed, and custom backends. Python `use_pool` is intentionally rejected. | Match Julia execution models while covering Python pool workflows without exposing Python-style pool controls. | `test/test_api_surface.jl`, `test/test_parallel.jl`, `test/test_static_sampler.jl`, `test/test_dynamic_sampler.jl` |
| Enum-like options | Python accepts strings such as `bound="multi"` and `sample="rwalk"`. | Julia accepts Symbols such as `bound=:multi`, `sample=:rwalk`, `parallel=:threads`, and `proposal_scheduler=:batch`; strings are rejected. | Julia-native option style and clearer API boundaries. | `test/test_api_surface.jl`, `test/test_parallel.jl`, `test/test_static_sampler.jl` |
| RNG keywords | Python uses `rstate` or NumPy random generators. | Julia uses `rng`, accepting an `AbstractRNG` or integer seed; `rstate` and `random_state` are rejected. Same-seed sample-by-sample equality with Python is not promised. | Use Julia RNG types and seeding semantics. | `test/test_api_surface.jl`, sampler reproducibility tests |
| Parallel reproducibility | Python pool trajectories depend on multiprocessing setup and NumPy RNG handling; `queue_size` can change the sequence of accepted proposals. | Dynesty.jl guarantees reproducibility for the same Julia seed, backend kind, backend configuration, thread count, `queue_size`, `ParallelPolicy`, and `proposal_scheduler`. Each proposal task receives an independent deterministic task RNG derived from the sampler RNG. It does not require identical trajectories across serial/threaded/distributed backends or with Python, and changing `queue_size`, parallel policy, or scheduler can change trajectories and performance. | Julia backends use Julia execution models and RNGs. | `test/test_static_sampler.jl` |
| Plotting | Python plotting targets Matplotlib. | Julia plotting uses RecipesBase-compatible recipes and optional Plots.jl smoke tests. | Avoid a heavy plotting dependency in the core package. | Stage 8 plot tests |
| Bound-plot live overlays | Python plotting is Matplotlib-oriented and can use Python result internals directly. | Julia `boundplot`/`cornerbound` draw saved bounds for static and dynamic `Results`; `show_live=true` is limited to static `Results` with reconstructable live points. | Dynamic `Results` do not carry enough final-live metadata for reliable backend-neutral live-point reconstruction. | `test/test_plotting.jl` |
| Persistence | Python sampler save/restore uses pickle and h5py-style evaluation history. | Dynesty.jl checkpoints use Julia Serialization `.jls`, results use JLD2 `.jld2`, and HDF5 evaluation history is optional via extension. Sampler checkpoints save lightweight backend configuration, parallel policy, proposal scheduler, counters, and stats, not worker process objects or pending threaded tasks; restore reconstructs the configured backend, and a distributed backend only uses worker IDs that are live in the current Julia session, otherwise falling back to serial execution for ordered maps. | Match Julia performance and package ecosystem while documenting archival limits. | Stage 1 persistence tests, `test/test_static_sampler.jl` |
| SupFriends centers | Python sampler sets `ctrs` externally before sampling; `SupFriends.update` does not assign centers directly. | Julia `update!(::SupFriends, points)` stores centers, matching `RadFriends` and making the bound self-contained. | Julia bounds should be valid immediately after update and still align with sampler behavior. | `test/test_bounding_friends.jl` |
| Static sampler bootstrap defaults | Python uses bootstrap expansion automatically for uniform ellipsoidal bounds. | Recursive multi-ellipsoid splitting and explicit ellipsoid `bootstrap > 0` are supported. The automatic uniform-bound default still maps to deterministic enlargement to keep default trajectories stable until that default is changed in its own compatibility stage. | Keep the static sampler runnable and reproducible while staged default-trajectory changes remain isolated. | `test/test_static_sampler.jl` |
| MultiEllipsoid split labels | Python uses SciPy `kmeans2` with major-axis endpoint seeds. | Julia uses Clustering.jl `kmeans!` with the same scaled endpoint seeds. Separable fixtures match numerically, but clustering label order is not public API. | Use Julia's scientific clustering dependency while preserving the recursive volume-reduction heuristic. | `test/test_bounding_unitcube_ellipsoid.jl` |
| Static sampler random trajectories | Python and Julia use different RNGs and proposal implementations. | Static sampler tests check invariants, result shapes, finite evidence traces, blobs, bound updates, and checkpoint restore rather than same-seed sample-by-sample equality. | Cross-language trajectory equality is not required by the migration policy. | `test/test_static_sampler.jl` |
| Results post-processing random trajectories | Python and Julia use different RNG implementations for jitter/resample realizations. | Deterministic helpers use Python fixtures; stochastic post-processing helpers use deterministic Julia seeds and invariant checks. | The migration policy requires reproducible Julia behavior, not cross-language same-seed trajectories. | `test/test_results_postprocess.jl` |
| Dynamic stopping Monte Carlo branch | Python draws seed sequences and realizations using NumPy. | Julia uses Julia RNGs and checks deterministic `n_mc=0` behavior against fixtures; Monte Carlo stopping checks use Julia reproducibility/invariants. | Same-seed cross-language equality is intentionally not required for random realization paths. | `test/test_dynamic_sampler.jl` |

Major public behavior changes must be discussed before implementation.
