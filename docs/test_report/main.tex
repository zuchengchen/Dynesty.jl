RSS 与 PSS 的口径不同。RSS（Resident Set Size）按进程统计驻留物理页，并在多进程场景中容易把共享页在多个进程上重复计数；因此 Python multiprocessing 的 process-tree RSS 往往显著偏高。PSS（Proportional Set Size）会按共享页的进程数分摊内存，更适合跨单进程多线程 Julia 与多进程 Python 的 process tree 比较。本报告在讨论内存时优先查看 PSS，同时保留 RSS 以便与常见系统工具口径对应。

从中位数看，cheap cost 下 Python wall time 为 3.82s，Julia 为 14.38s；medium cost 下 Julia 为 14.39s，Python 为 37.34s；heavy cost 下 Julia 为 14.90s，Python 为 43.50s。PSS 中位数方面，Julia 三组约为 942,168 KB、901,831 KB、922,647 KB；Python 三组约为 213,612 KB、218,819 KB、218,415 KB。RSS 中位数方面，Python 三组约为 1,602,416 KB、1,608,788 KB、1,606,440 KB，受 multiprocessing 共享页重复计数影响，不能单独作为跨执行模型内存优劣结论。

\section{真实空气质量 PE Benchmark}

新增 benchmark 位于 \pathcode{examples/air_quality_likelihood.jl}、\pathcode{examples/air_quality_pe_julia.jl}、\pathcode{examples/air_quality_pe_python.py}、\pathcode{benchmark/air_quality_pe_compare.jl} 和 \pathcode{benchmark/air_quality_corner_overlay.py}。模型是 deterministic synthetic air-quality inverse problem：56 个 PM2.5 sensors、48 个 hourly time slices、固定 synthetic source location，反演 \code{source_strength}、\code{background_pm25}、\code{wind_decay}、\code{diffusion_length}、\code{global_bias} 和 \code{log_noise_scale} 共 6 个参数。观测由固定 seed 生成，不包含真实个人或敏感数据；代码和 metadata 明确说明该 synthetic model 不用于实际环境监管决策。

likelihood 是唯一 canonical implementation：\code{examples/air_quality_likelihood.jl} 中的 Julia \code{AirQualityPE.air_quality_loglikelihood}。Julia \code{Dynesty.jl} runner 直接调用该 Julia likelihood；Python dynesty runner 从只读 \pathcode{../dynesty/py} 导入 Python dynesty，并通过 Python-Julia bridge 调用同一个 Julia likelihood，没有实现 Python duplicate likelihood。当前环境没有 \code{juliacall}，可用 fallback 为 PyJulia；由于本机 Python/PyCall 组合需要 \code{compiled_modules=false}，runner 在 metadata 中记录 \code{bridge_kind=pyjulia}、\code{bridge_compiled_modules=false}，并且必须先初始化 Julia bridge 再导入 \code{numpy}/\code{dynesty}，否则 PyJulia 会在加载 \code{PyCall} 时失败。

本节结果来自 documented scaled formal run。配置为 \code{nlive=500}、\code{dlogz=0.08}、\code{repeats=2}、\code{queue_size=31}、Julia \code{--threads=31}、Python \code{nproc=31}，并设置 \code{OPENBLAS_NUM_THREADS=1}、\code{OMP_NUM_THREADS=1}、\code{MKL_NUM_THREADS=1}。likelihood calibration 选择 \code{work_repeats=64}，Julia direct median 为 \code{0.00906s}，满足 0.005--0.02 秒目标区间；Python bridge median 中位数为 \code{0.00897s}。全局系统缺失 \code{/usr/bin/time}，因此本次 run 在临时 mount namespace 中把编译的 GNU time 暴露为 \code{/usr/bin/time}；benchmark metadata 记录 \code{time_command_available=true}、\code{used_usr_bin_time=true}。Linux \code{/proc} process-tree monitor 同时记录了 RSS/PSS peak，采样间隔为 \code{0.2s}。

\section{结论与剩余风险}

本次常规 Julia package/test suite 全量通过，且 \code{test/runtests.jl} 覆盖了 parallel、examples、Python fixtures、plotting helpers 和 Aqua quality checks。benchmark runner include check 与 Python \code{py_compile} 也通过。formal benchmark 没有重跑，而是复用已有 \code{examples/output/parallel_cost_compare} 输出；审计确认该输出完整、一致，并且记录了 \code{/usr/bin/time -v} 与 process-tree RSS/PSS 指标。

posterior overlay 图在 cheap、medium、heavy 三种 cost 的 3 次 repeat 上全部生成成功；代表图和附录列表显示 Julia/Python 后验叠加比较已经可用于可视化层面的一致性检查。新增空气质量 PE documented scaled formal 也已生成 2 张 Julia/Python posterior overlay，且 Python dynesty 确认通过 PyJulia bridge 调用同一个 Julia likelihood。性能方面，当前 parallel cost benchmark 下 Python multiprocessing 在 cheap likelihood 上 wall time 更低；当 likelihood cost 提升到 medium/heavy 后，Julia threaded 的 wall time 中位数更低。空气质量 benchmark 中 Julia median wall time 为 \code{25.10s}，Python PyJulia path median wall time 为 \code{31.81s}，但该结论包含 PyJulia initialization、31 个 worker runtime 和当前 synthetic model 配置。

剩余风险包括：nested sampling 与 posterior visualization 受随机性和有限 repeat 数影响；性能结论依赖 AMD Ryzen 9 5950X、当前 Julia/Python/BLAS/线程设置和操作系统调度；Python multiprocessing memory accounting 的 RSS/PSS 解释依赖 Linux \code{/proc} 口径；空气质量 benchmark 是 prompt 允许的 documented scaled formal，而不是 800--1500 nlive、3 repeats 的完整目标规模；PyJulia fallback 使用 \code{compiled_modules=false}，初始化开销和 worker memory 与 \code{juliacall} 或 \code{python-jl} 路径可能不同；扩展测试 flags 未在本次默认报告中打开；future benchmark 输出若更新，应重新运行 \code{docs/test_report/generate_assets.py} 并重新编译 PDF。
